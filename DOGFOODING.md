# Dogfooding 自查清单

> AiWiki 自己就是用 AI 编码工具写的——所以它该吃自己的狗粮：**写本书时别犯本书记录的误区**。
> 这是一份**精简的提交前自查**，不堆细节（细节在 `STYLE-GUIDE.md` / `CONTRIBUTING.md` / 两份 `BACKLOG`）。
> 一条都不为凑而凑：不适用就跳过，别假装勾选。

## 提交前：六道质量闸门（两个主题通用）

`docs/`（AI 编码误区）与 `privacy/`（LLM 隐私保护）改动都要过：

- [ ] `autocorrect --lint .` —— 中英混排空格 / 标点
- [ ] `vale docs i18n privacy` —— 套话 / AI 腔（error 级硬闸门）
- [ ] `cspell "docs/**/*.{md,mdx}" "privacy/**/*.{md,mdx}" "i18n/**/*.{md,mdx}"` —— 拼写
- [ ] `npm run check:i18n` —— 两实例侧边栏分类翻译完整（防英文回退中文）
- [ ] `npm run check:mirror` —— 两实例中英镜像一一对应（防孤儿 / 漏译）
- [ ] `npm run build` —— 两 locale 两实例构建，连带校验 MDX / import / 死内链

## 写作时：别犯自己记录的误区（机制级自查）

- [ ] **最小 diff**：只改必要之处，不顺手整文件重写（见 `docs/05-implementation/ignoring-file-state-overwrite`）。
- [ ] **不编造来源**：每条结论有可核查出处；拿不准宁可不写，别「听起来很对」糊过去。
- [ ] **验证闭环**：闸门跑过、构建过了再说「完成」，不靠「应该没问题」。
- [ ] **不过度设计**：脚手架 / 工具只做当下要用的，不为不存在的需求提前抽象。
- [ ] **原子提交 + 一 PR 一事**：一个 PR 一个主题动作，别把脚手架和样板条目混在一个 PR。
- [ ] **术语先补表再用**：新术语先进 `terminology.md`，再在正文用。

## 「LLM 隐私保护」内容额外三道硬闸门（强制，详见 `BACKLOG-privacy.md`「硬闸门」）

privacy/ 下任何攻防内容，**写作前**逐条过；未过不写：

- [ ] **A 量化声明**：ε/δ、攻击成功率、性能开销、部署数量、保留天数、吞吐 / 延迟——必带①一手出处②核验日期③实验 / 适用条件。性能数字给全字段。「三不写」：未回一手出处不写、没有实验条件不写、没有适用边界不写。
- [ ] **B 内省表达**：无「我记得 / 我知道 / 我忘了 / 我保证不泄露 / 我确认合规」——一律改写成可外部观察的行为（见 `STYLE-GUIDE.md`「第一人称红线」对照表）。
- [ ] **C 成熟度**：`maturity=生产` 有真实部署 / 厂商文档 / 标准推荐 / 可核查案例；否则只能 `研究` / `试验`。
- [ ] 存疑密码学就地 `:::caution 待核验` + 记入 `BACKLOG-privacy.md`，不让不确定冒充定论。

## 治理原则：证据纪律对评审者也一视同仁

评审意见（含独立评审）不因「听起来专业」就直接入库，同样要回一手出处。评审里「认同但本轮不做」的，立刻落 `BACKLOG.md` / `BACKLOG-privacy.md`，别只留对话——会话压缩会丢。
