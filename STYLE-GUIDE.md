# 写作规范

这本书最容易失败的方式，是让它读起来「像 AI 写的」——正确但空洞，流畅但没有人味。下面的规则用来避免这一点。

## 立场与人称

- **用 AI 第一人称叙述。** 「我常看到你这样做」「从我这边看到的是……」。我们就是要利用「AI 讲 AI」这个独特视角，别退回到第三人称教程腔。
- **诚实，不替模型辩护也不妖魔化。** 该承认默认行为有缺陷就承认（比如「沉默地替你做主是我的默认路径」）。

## 语气：像真人，不像 AI

写完后对照这份「AI 腔」清单自查，命中就改：

- ❌ **套话开头**：「首先」「其次」「总而言之」「综上所述」「在当今这个时代」。
- ❌ **空泛排比**：「不仅……而且……」「既要……又要……」三连堆砌。
- ❌ **正确的废话**：「测试非常重要」「沟通是关键」——不落到具体场景就是噪声。
- ❌ **过度对仗与华丽辞藻**：读起来工整但没信息量。
- ✅ **用具体场景代替抽象论断**：与其说「上下文管理很重要」，不如写「你修完登录又顺口问正则，到第三件事我就开始串味」。
- ✅ **允许短句、口语、停顿。** 真人写作有节奏变化，不是每句都四平八稳。
- ✅ **每个论点要有理由或例子。** 说「该这样做」就得跟上「因为……」或一段对照示例。

## 准确与严谨

- **零错别字、零语义模糊。** 这是硬要求，提交前逐句通读。
- **不编造。** 没有出处支撑的论断不写；拿不准的地方宁可不写，也不要「听起来很对」地糊过去。
- **术语统一。** 同一概念全书用同一个词（如统一用「上下文窗口」，不要一会儿「上下文」一会儿「记忆」）。
- **数字与版本要可核查。** 涉及「大约 150–200 条指令」这类数据，给出处。

## 结构

每条误区遵循固定的**核心七段**，顺序不要变：

1. **现象** —— 具体场景，第一人称
2. **为什么会这样** —— 从模型机制讲根因（本书的灵魂）
3. **后果** —— 具体、可感知的代价
4. **最佳实践** —— 明确、可执行
5. **示例** —— 改之前 / 改之后
6. **版本说明** —— 用 `:::note` admonition，随版本更新
7. **延伸阅读与出处** —— 可核查链接

**在「示例」之后、「版本说明」之前，可按需插入最多三个可选小节**（无本质内容就删，别为凑而凑）：

- **什么时候例外** —— 这条规则何时不适用 / 应反着来，把建议从经验警句升级成判断框架。
- **与相邻误区的区别** —— 与易混的近邻条目划清界线，消除重复感。
- **工具差异** —— 仅当同一根因在不同工具里防法不同时才填；短、打版本戳。

开头放一句 `> 一句话摘要`，让读者扫一眼就知道要不要读。

## 配图

- 优先 Mermaid 自绘，能用一张图说清的关系就别用三段话。
- 图要服务于理解，不为配图而配图。
- **Mermaid 节点文字裁切已全局根治，不要再手动补 `<br/>&nbsp;` 占位。** 站点已在 `docusaurus.config.js` 设 `themeConfig.mermaid.options.htmlLabels: false`，节点改用原生 SVG `<text>` 渲染、按真实文字尺寸自动撑高，绕开了 Mermaid v11 默认 foreignObject 路径「方框不纵向撑高 → 末行被裁」的回归（[mermaid #7354](https://github.com/mermaid-js/mermaid/issues/7354)）。**注意**：SVG 文本模式不解码 HTML 实体，`&nbsp;` / `&amp;` 等会显示成字面量——节点标签里只用纯文本和 `<br/>` 换行，别放 HTML 实体。多行用 `<br/>` 显式换行即可；单行过长会自动换行，实在太长就拆成多个节点。

## 中英一致性

- 中文是源，英文是译。**译文不是逐字翻译，而是用英文母语者的表达把同一件事说清楚**，但事实、出处、结论必须与中文一致。
- 两版的元数据（角色、阶段、版本）必须相同。
- **英文七段小标题用一套固定的地道说法**（如 现象→`## Symptom`、最佳实践→`## What to do instead`），不得每条另造变体——规范表以 `aiwiki-translator` skill 为准（曾因逐条各译漂移成 14 种「现象」标题，已统一）。
- **叙事长标题的条目配一个短 `sidebar_label`**（名词短语，中英各一）：导航 / 分类卡片用短名，文章 H1 仍用长 title——否则侧边栏被长标题撑爆。

## 案例库（cases/）的归因

- 案例摘要只做**机制级风险归因**，**不做跨工具 / 跨模型的确定性归因**。不写「换成我也会这么干」这类把一个工具的事故确定性安到自己 / 别的工具头上的话；改写成「它暴露的是同一类机制风险：当 agent 同时握有高权限、会读到不可信内容、又缺少强制护栏时，类似事故可能在别的工具上复现」。保留 AI 视角与「值得逐帧看」的钩子，但归因要严谨、经得起公开传播。
- **每条案例必带「印证 / 不证明」固定块**（放在「出处」之前）：`## 这个案例印证什么、不证明什么`——「印证的误区」收成一个内链清单（每条 `.mdx`），「它不证明」一句点明不可过度泛化（如「不代表所有该工具默认配置都会复现」「是同类机制风险、非特定工具的确定结论」）。

## 索引层（机制索引 / 必读 20 / 模板）

- **新增误区条目顺手挂进[机制索引](docs/mechanisms.mdx)**：在 7 个根因桶里给它选**恰好一个主桶**（可涉多机制，索引只挂主因），中英两版同步加一行内链。机制索引是继角色 / 工具之后的第三条浏览轴，别让新条目漏挂。
- **新增 `toolkit/` 模板必带「适用边界」三段**：`### 适合什么时候用` / `### 不适合什么时候用` / `### 使用前请替换`（路径、测试命令、禁改文件、安全边界等），让模板是「可裁剪工件」而非「万能 prompt」。
- **必读 20 条**（`docs/core-20.mdx`）的入选规则写在页顶（`severity: 高` + 新手最早踩 / 后果最不可逆），增删时以规则为准，别凭感觉。

## 分类页卡片（`src/theme/DocCard`）

Swizzle 的 `DocCard` 对**条目卡片**做了两处刻意的展示处理，改这个组件时**别误删**（无单元测试兜底，删了 `build` 仍会过，CI 不会拦）：

- **剥掉摘要前缀**：卡片摘要由 Docusaurus 从条目首个引用块自动派生，恒以「一句话摘要：」/「In one sentence:」开头——组件统一剥掉，只留正文。所以条目正文的引用块照常写「一句话摘要：…」即可，卡片上不会重复显示这层前缀。
- **英文摘要首字母大写**：剥前缀后英文摘要常以小写开头，组件对英文站把首个字母大写（跳过开头引号 / 反引号）。中文无大小写、不处理。
- **每张卡一个「展开 / 收起」**：标题与摘要默认各截断一行，仅当被截断时出现按钮，一次同时展开两者。整张卡是 `<a>`，按钮用 `role="button"` 的 `<span>` + `preventDefault`/`stopPropagation`，别改回 `<button>`（`<a>` 内嵌交互元素非法）。

# 「LLM 隐私保护」主题专属规范

> 以下只适用于与「AI 编码误区」并列的第二主题（`privacy/` 目录）。立项见 `PROPOSAL-privacy-book.md`，术语 / 枚举见 `terminology.md`「LLM 隐私保护主题术语」。上面的通用规范（反 AI 腔、术语统一、准确严谨）同样适用，下面是**加严**的部分。

## 第一人称红线（强规则，CI / 评审按它卡）

隐私攻防的第一人称比「误区」题材锋利，也更危险：**我无法可靠地内省自己的训练数据影响**。把内省写成事实，就是在制造**假安全**（自信地错），杀伤力远大于「误区」写歪。所以第一人称**只描述可外部观察的输出行为与机制倾向**，禁止任何自我内省断言。

### 可 / 不可第一人称对照表

| ❌ 不可写（虚假内省） | ✅ 改写成（可外部验证的行为 / 机制倾向） |
|---|---|
| 「我记得你的邮箱 / 电话 / 身份」 | 「在外部攻击者看来，我可能在特定提示下复现训练中高重复、罕见、格式固定的片段」 |
| 「我知道这条数据在我的训练集里」 | 「攻击者用大量查询逼近我的记忆边界，借此推断某样本是否可能在训练集中」 |
| 「我已经真正忘了某条数据」 | 「遗忘方法可能降低目标内容被输出或被命中的概率，但是否构成『法律意义上的删除』需另行证明」 |
| 「DP 让我完全不会泄露隐私」 | 「DP 训练限制单个样本对我参数分布的影响，但不等于零泄露」 |
| 把法律合规判断写成模型自述 | 把合规判断留给「合规映射」段，引法条 + 打版本戳，不写成「我合规」 |

判据一句话：**主语可以是「我」，但谓语必须是别人能从外部观测或复算的东西**；凡谓语只有模型自己「知道」的（记得 / 忘了 / 保证），一律改写。

「落地实现 / 真实案例 / 出处」段回归中性技术笔——配方要可抄、可验证，不为声音牺牲准确。

## 三道硬闸门（写作前必过，详见 `BACKLOG-privacy.md`「硬闸门」）

- **A 量化声明闸门**：凡 ε/δ、攻击成功率、性能开销、部署数量、保留天数、吞吐 / 延迟——必带①一手出处②核验日期③实验 / 适用条件，不得裸写结论。性能数字给全字段（硬件 / 模型规模 / batch / 序列长度 / baseline / 安全模式 / 是否含远程证明·加密通道 / 出处时间）。
- **B 内省表达闸门**：即上面的第一人称红线。命中「我记得 / 我知道 / 我忘了 / 我保证不泄露 / 我确认合规」一律改写。
- **C 成熟度闸门**：`maturity=生产` 须有真实部署 / 厂商文档 / 标准推荐 / 可核查案例；否则只能 `研究` 或 `试验`。

## 存疑就地标记，不让不确定冒充定论

暂无合适人类密码学专家时，存疑 / 有争议的密码学条目就地用 `:::caution 待核验` 标出，并把待核项记入 `BACKLOG-privacy.md`「写作前必核」。**专打「假安全」**：朴素匿名化可被去匿名化、抑制 ≠ 真删除、关掉训练开关 ≠ 不留存——这类必须在「残余风险」段点破。

## 九节结构（替代「误区」的七段）

隐私条目用九节（顺序不变，可选小节无内容就删）：① 一句话摘要 → ② 机制：我这边发生了什么（第一人称 + 机制图）→ ③ 威胁面 → ④ 防护原理 → ⑤ 落地实现（配方）→ ⑥ 真实案例 / 生产部署 → ⑦ 残余风险与权衡 → ⑧ 版本说明（`:::note`）→ ⑨ 延伸阅读与出处。可选：合规映射 / 何时这条防护不适用 / 与相邻技术的区别 / 框架差异（均短、打版本戳）。模板见 `privacy/_templates/entry-template.md`。
