## 0. 一句话立项

在 AiWiki 百科下新增一个**面向 LLM 隐私保护工程师**的主题（工程向参考手册形态）：从 LLM 发展史讲清「隐私攻防机制如何演进、如何落地」，原理 + 可实现配方 + 真实案例三件套，随模型迭代持续更新，中文为源、英文镜像。**截至 2026-06，我尚未看到主流公开资源同时满足：LLM 专门、工程可落地、持续更新、真实案例、证据可核查、中文源 + 英文镜像、统一叙事**——这正是 AiWiki 既有流水线能精准卡住的空位。（带时间戳 + 判定条件，比「没有任何」更经得起反例。评审 R1/R2）

## 1. 背景与定位

- **是什么**：AiWiki 百科下一个「从 AI 第一人称视角」讲 LLM 隐私保护的主题（工程向）。和现有主题共用「机制优先、第一人称、证据可核查、随版本更新」的内核，但**领域、单元、结构都换**。
- **读者**：LLM 隐私保护工程师，以及 ML 工程师、AI 安全工程师、隐私 / 合规工程师。比「AI 编码误区」主题的受众更窄、更专——**流量天花板低，但深度黏性强**。
- **与其他主题的关系**：**并列的两个主题**。「AI 编码误区」讲「用 AI 编码工具时人最容易犯的错」；「LLM 隐私保护」讲「LLM 系统里隐私如何被泄露、如何被保护」。两者都属 AiWiki 这个「从 AI 视角写的工程百科」。
- **为什么是现在**：LLM 隐私是**月级演进**的活领域（机密推理、可扩展遗忘、DP 联邦生产部署接连落地），而现有资料要么过时、要么是论文堆。持续更新的工程手册正当其时。

**关键观察**

- 最接近「可读手册 + 持续更新 + 工程向」的是 LLM **安全 / 红队**手册——但它们主攻 prompt injection、越狱、对抗攻击，**隐私（记忆抽取、DP、遗忘、机密推理、合规）只占一两节**。
- 隐私一侧的资源清一色是**论文清单或学术综述**，没有「读得下去、抄得动手」的手册。
- **中文世界更空**：只有 CSDN 博客、几篇期刊综述（《计算机研究与发展》《软件学报》）、NTU/NUS 的全栈 LLM Safety 综述，**没有一本面向工程师的大模型隐私专著或在线手册**。

## 3. 缺口：为什么这个位置是空的

- **论文清单有广度没「人味」**：`Awesome-LM-SSP` 收 723 篇隐私论文，但它是导航，不教你「DP 微调怎么做、ε 怎么选、遗忘怎么验证」。
- **综述有深度但是快照**：写完即开始过时，学术腔、无工程配方、常在墙后。对月级演进的领域天然吃亏。
- **标准权威但碎**：OWASP / NIST 各管一段，合规导向，不成学习路径，不深入实现。
- **厂商指南实用但锁厂商**：教你用它的产品，不讲机制，换一家就失效。
- **通用 PPML 书过时**：DP / 联邦 / HE 讲得好，但成书于 LLM 大爆发前，记忆抽取、系统提示词泄露、agent 串味、机密推理这些 **LLM 特有面**基本没有。

**没有人把这些缝成一条线**：从机制讲清「模型到底怎么记住、怎么泄露、怎么被迫遗忘」，再给可落地实现 + 真实案例 + 随模型更新。这条线就是本书的位置。

## 4. 值不值得做（优缺点 + 结论）

**做的理由**

- 空位真实，且与 AiWiki 资产高度契合——流水线、双语、证据 / 版本戳、案例库、质量闸门**可几乎原样复用**，边际成本低。
- **第一人称 AI 视角在隐私题材上比在「误区」题材上更锋利**：隐私攻防本质就是「模型内部发生了什么」，让模型现身说法是任何论文清单 / 综述 / 厂商文档都给不了的叙事——**天然护城河**。
- 持续更新是该领域刚需，而 `aiwiki-research-refresh` 月度流水线天生干这个。
- 中文市场近乎空白，双语策略一鱼两吃。

**慎做的代价（诚实话）**

- **准确门槛陡升**：DP 的 ε/δ、HE/MPC 的安全假设、攻击成功条件——写错会让工程师做出**假安全**系统，杀伤力远大于「误区」写歪。需更硬的出处纪律，最好引入技术评审。
- **维护是长期承诺**：第二条要月度喂养的产线，人力翻倍。
- **体裁与现有声音不完全兼容**：「误区 / 第一人称」要改造成「技术 / 攻防对」单元（见第 7 节）。
- **受众更窄**：流量天花板低于「AI 编码误区」，但深度黏性更强。

**结论：做，但定位「精而深的参考手册」，不追广度铺条数。** 宁可 30 条扎实到能照着实现，也不要 100 条综述式摘抄。

## 5. 差异化策略（护城河）

1. **模型第一人称讲隐私**（**可读性 + 品牌识别**差异点，非可信度护城河——见本节末校准）：把攻防写成「我的自述」——我如何记忆、回吐、被抽取、被推断成员、被迫遗忘。我还没见到谁这么写隐私。
2. **历史演进为阅读主线**：把零散技术缝成一条故事线（见 6.1）。
3. **原理 → 可落地实现 → 真实案例三件套**：每条都要有机制图 + 代码 / 配置配方 + 一个真实事故或真实部署。
4. **持续版本戳 + 月度刷新**：每条打「截至 202X-XX」，模型一代新攻防就回填——把「会过时」从弱点变成卖点。
5. **证据分级 + 不编造铁律**：沿用 evidence 标签（官方文档 / 研究支持 / 安全报告…），拿不准不写，对内容农场形成信任差。
6. **可操作的索引轴**：除历史主线，另做技术索引 + 威胁→缓解矩阵（见 6.2 / 6.3），让人能「查」而不只是「读」。

> **护城河校准（评审 R1/R2）**：分三层——**品牌识别 = 第一人称**（入口差异，会被模仿，**非防守壁垒**）；**可信度护城河 = 证据纪律 + 工程闭环 + 月度刷新**；**内容壁垒 = 攻击 / 防护 / 残余风险 / 成熟度 / 案例 / 配方**。第一人称负责把人领进门，证据纪律与工程闭环负责让人留下、让对手抄不走。

## 6. 内容架构（决策 ③A：历史主线 + 技术索引双轴）

延续第一本书「阶段为主线 + 角色 / 工具 / 机制为交叉索引」的成熟做法：**历史卷做阅读路径，技术板块 + 威胁矩阵做查询轴**。同一条目同时挂在「它属于哪一卷」与「它属于哪个技术板块」。

### 6.1 阅读主线：按 LLM 演进史分卷

- **卷一 · 隐私根基（前 LLM 与判别式时代）**
  嵌入 / 表示泄露与文本去匿名化；成员推断攻击（MIA）的起源（判别式模型时代）；隐私保护计算地基——差分隐私（DP）、联邦学习（FL）、同态加密（HE）、安全多方计算（MPC）、可信执行环境（TEE）的原理与威胁模型。**作用**：把后续各卷复用的密码学 / 统计学地基一次讲清。
- **卷二 · 记忆与抽取（生成式预训练时代，GPT-2/3）**
  逐字记忆与量化记忆；训练数据抽取攻击（Carlini 等 2021/2022 是里程碑）；记忆的影响因素（重复、规模、上下文）；防护——去重、DP 预训练、记忆审计。**这是旗舰卷**（故事性最强、证据最硬）。
- **卷三 · 对话大模型时代（ChatGPT 起）**
  PII 回吐（2023「无限重复某词」泄露真实邮箱 / 电话事件）；系统提示词提取与上下文泄露（prompt extraction / inversion）；DP 微调实用化（DP-SGD、PEFT + DP）；成员推断在 LLM 上的演化。
- **卷四 · RAG 与 Agent 时代**
  检索泄露（向量库 / 知识库跨租户、跨用户）；跨会话 / 跨用户记忆串味；工具调用与 agent 的隐私外泄（私有数据发到外部）、致命三件套的隐私视角复现；多 agent 协作的数据边界。
- **卷五 · 当下前沿与工程落地**（评审 R1：本卷拆「① 私有计算与机密推理 ② 可验证删除与机器遗忘 ③ 生产级 DP·FL 部署」三小节）
  机密推理（TEE / GPU 机密计算、远程证明；**开销高度依赖 workload，不给单一乐观数、须带条件**——有研究报 H100 机密计算 throughput penalty 约 4–8%，也有在模型换入换出等场景观察到低 45–70%，这些数连同原「95–99%」均入「写作前必核」）；加密推理（HE/MPC 的现状与约万倍开销现实）；可扩展机器遗忘与被遗忘权工程化（抑制 vs 真删除、验证难题）；DP 联邦生产部署（Google Gboard，DP 模型数以一手论文为准；Apple 本地 DP）。
- **卷六（活页）· 治理与合规映射**
  GDPR Art.17（被遗忘权）对训练模型的工程含义；EU AI Act 训练数据透明度；NIST SP 800-226 / AI 600-1；OWASP LLM Top 10（LLM02）到工程动作的映射；数据边界策略与厂商数据使用条款核查（如 Anthropic 默认不用 API 数据训练）。

### 6.2 技术索引轴：12 板块（查询用）

1. 记忆与训练数据抽取
2. 推断类攻击（成员推断 MIA / 模型反演 / 属性推断 / 属性泄露）
3. 模型抽取与窃取
4. 差分隐私（预训练 / 微调 / PEFT / RLHF / 推理 / 联邦各场景）
5. 联邦学习与安全聚合
6. 机器遗忘与被遗忘权
7. PII 检测 · 脱敏 · 匿名化 · 合成数据
8. 隐私保护计算（HE / MPC / TEE 机密计算）
9. RAG 与 Agent 隐私
10. 系统提示词与上下文泄露
11. 推理服务期隐私（prompt PII / 日志留存 / 数据边界 / 厂商数据使用）
12. 治理与合规

### 6.3 威胁 → 缓解矩阵（查询轴，类比现有 `/threat-model`）

一张「隐私威胁维度 ×（攻击 / 缓解 / 残余风险 / 对应条目）」的策展页：训练数据泄露、推理输入泄露、跨用户串味、模型本身被窃、合规删除诉求……每个维度链到对应条目与案例。顶部放一份「最小隐私基线」速用清单（仿现有 threat-model 顶部做法）。

## 7. 条目单元与写作声音（决策 ②A）

### 7.1 单元：从「误区」到「机制 / 攻防对」

第一本书的基本单元是「**误区**」（pitfall）。本书换成「**机制 / 攻防对**」：一种隐私机制、一类攻击、或一对「攻击 ↔ 防护」。`terminology.md` 需为本书补一段术语（ε/δ、DP-SGD、PATE、FHE、MPC、TEE、MIA、unlearning、secure aggregation…）。

### 7.2 条目结构（在七段精神上改造为九节）

1. **一句话摘要**（`> 一句话摘要：…`，沿用）
2. **机制：我这边发生了什么**——第一人称 + 机制图（本书的灵魂，对应原「为什么会这样」）
3. **威胁面：如何被利用 / 你如何被泄露**
4. **防护原理**
5. **落地实现（配方）**——代码 / 配置 / 参数，回归中性技术笔
6. **真实案例 / 生产部署**——映射到案例库
7. **残余风险与权衡**——ε、性能、效用代价；点破「假安全」
8. **版本说明**（`:::note`，打版本戳）
9. **延伸阅读与出处**

可选小节（无内容就删）：**何时这条防护不适用** / **与相邻技术的区别** / **框架差异**（Opacus、TF-Privacy、PySyft、OpenDP… 类比现有「工具差异」，短、打戳）。

### 7.3 第一人称声音的边界（②A 的纪律）

- **第一人称红线（强规则，评审 R1/R2，非建议）**——第一人称只描述**可外部观察的输出行为与机制倾向**，**禁止任何自我内省断言**（模型无法可靠内省自己的训练数据影响，写成内省事实即失准，是「假安全」/「自信地错」）：
  - ❌ 禁止：「我知道这条数据在训练集中」/「我记得你的邮箱·电话·身份」/「我已经真正忘了某条数据」/「DP 让我完全不会泄露隐私」/ 把法律合规判断写成模型自述。
  - ✅ 可写：「在外部攻击者看来，我可能在特定提示下复现训练中高重复、罕见、格式固定的片段」/「DP 训练的目标是限制单个样本对参数分布的影响，但不等于零泄露」/「遗忘方法可能降低目标内容被输出或被命中的概率，但是否构成『法律意义上的删除』需另行证明」。
- **落地实现 / 案例 / 出处**段回归中性技术笔——配方要可抄、可验证，不为声音牺牲准确。
- 此红线 + 「可 / 不可第一人称」对照表写进隐私版 `STYLE-GUIDE`（强规则，评审与 CI 按它卡），避免跨条目漂移（第一本书已踩过「逐条各译漂移」的坑）。

### 7.4 frontmatter 轴（起点，细节留到脚手架阶段）

- **复用**：`title` / `slug` / `sources` / `evidence` / `tags` / `sidebar_position`。
- **调整**：`roles` 改义为受众（隐私工程师 / ML 工程师 / 安全工程师 / 合规）；`phase` 复用为「**卷**（era）」轴。
- **新增**：`technique`（12 板块之一）、`attack_surface`（攻击面）、`maturity`（技术成熟度：研究 / 试验 / 生产——防止把研究级方案当可部署）。
- `severity` 改义为「**隐私风险等级**」，并考虑与「数据敏感度」分离（待决，见第 13 节）。

## 8. 站点技术架构（多实例 docs，待脚手架阶段实施）

要做成「并列两本书」而非「一本书两部分」，用 Docusaurus **多实例 docs 插件**：

- 现有 preset `docs` 保持为**第一本书**（`routeBasePath: '/'`）。
- 新增一个 `@docusaurus/plugin-content-docs` 实例：`id: 'privacy'`、`path: 'privacy'`、`routeBasePath: 'privacy'`、独立 `sidebars-privacy.js`。
- i18n 英文镜像走 `i18n/en/docusaurus-plugin-content-docs-privacy/current/`。
- navbar 加「书籍切换」（下拉或两个顶层入口），各自指向对应实例侧边栏。

## 10. 质量与准确性纪律（比误区书更严）

- **密码学 / DP 事实必带一手出处 + 版本戳**：ε/δ 取值、性能开销（如 HE 约 10⁴× 开销、GPU 机密计算约 95–99% 原生性能）、攻击成功条件，不靠记忆。
- **区分「研究结果」与「生产可用」**：用 `maturity` 字段标清，别让研究级方案读起来像能直接上线。
- **量化安全声明需 ≥2 个独立出处**；涉密码学正确性的条目，最好过技术评审人。
- **专打「假安全」**：朴素匿名化可被去匿名化、unlearning 抑制 ≠ 真删除、关掉训练开关 ≠ 不留存——这类「以为安全其实没有」必须在「残余风险」段点破。

## 11. 风险与边界（诚实话）

- **别写成安全 / 红队书**：与 OWASP / 红队手册划界——它们管「被攻破」，本书管「数据 / 身份不被泄露与可被遗忘」。重叠处（提示注入致数据外泄）做交叉链接而非重复。
- **密码学部分最易写错**（见第 10 节）——这是本书的存亡风险。
- **维护翻倍**：两条产线都要月度喂养，需评估人力。
- **受众窄**：流量低于第一本书，靠深度与持续更新立身。
- **GitHub 连接波动**：本地写作 / 构建不受影响；开 PR 阶段若代理掉线，先在分支备好改动再推。
