# 按角色浏览

每条误区都标注了它最相关的角色。挑你的角色，直接看与你日常最贴近的那些坑——不用从头翻整本书。

## 🧭 项目经理 / 产品

需求有没有被真正听懂、可行性会不会被我的乐观带偏、「demo 能跑」是不是被当成了通过验收。你最该盯的是「我把模糊当明确、把『能做』当『可行』」这一类。

[查看「项目经理」相关的全部条目 →](/tags/项目经理)

## 🏗 架构师

选型的权衡有没有摊开、我会不会过度设计、跨模块改动时的重复逻辑与幻觉 import，以及为什么关键决策必须落成 ADR（否则下个会话的我就把它推翻了）。

[查看「架构师」相关的全部条目 →](/tags/架构师)

## ⌨️ 工程师

坑最密的地方。上下文管理、幻觉 API 与版本漂移、过度编辑、覆盖你的改动、边界条件，以及怎么用测试给我建一个验证闭环。

[查看「工程师」相关的全部条目 →](/tags/工程师)

## ✅ 测试工程师

我「看起来对」不等于「真的对」。别让我只测 happy path、别让我为了通过去改测试或造假断言、回归别漏——把验证变成能自己跑、能给红绿信号的闭环。

[查看「测试工程师」相关的全部条目 →](/tags/测试工程师)

## 🚀 运维工程师

权限与发布面的事。最小权限、别一上来全开、MCP 会扩大提示注入面、CI/CD 的权限边界、发布前的安全与敏感数据。

[查看「运维工程师」相关的全部条目 →](/tags/运维工程师)
