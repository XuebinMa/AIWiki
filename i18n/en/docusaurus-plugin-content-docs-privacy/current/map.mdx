# Privacy map

The sidebar organizes by **volume** (the reading spine). This page adds two **query axes**: find entries by **technique board**, or look up by **threat → mitigation** — "which entries cover this leak surface, and what residual risk remains."

:::note On coverage
This theme is deep-not-wide and doesn't chase entry count (see [PROPOSAL](https://github.com/xuebinma/AIWiki/blob/main/PROPOSAL-privacy-book.md)). **All 14 technique boards now have at least one shipped entry, or are covered "running through every entry"** — from here we deepen rather than pad the board count. If a `_planned_` marker ever reappears, it's a gap called out on purpose, so "looks complete" doesn't hide a blank (exactly the kind of false security this book argues against).

## Threat → mitigation matrix

Look up by "the leak surface you worry about": which entries cover it, and what they **don't** solve.

| Threat (what you worry about) | Entries that cover it | Residual risk / what's not solved |
|---|---|---|
| Private data **memorized and reproduced verbatim** | [Training-data extraction](./02-memorization-extraction/training-data-extraction.mdx) | dedup ≠ elimination, DP ≠ zero leakage, output filtering is cat-and-mouse |
| Deciding **whether someone is in the training set** (membership itself is sensitive) | [Membership inference](./01-foundations/membership-inference.mdx) · [DP fine-tuning](./03-conversational-llms/dp-fine-tuning.mdx) | average accuracy hides the tail, reducing overfitting has no formal guarantee, watch ε |
| Exercising the **right to be forgotten / deletion** while the model still remembers | [Machine unlearning](./05-frontier-deployment/machine-unlearning.mdx) | output suppression ≠ true deletion, approximate unlearning unguaranteed, verification is hard |
| **Vector store / retrieval** leakage: embedding inversion, cross-tenant bleed | [RAG retrieval leakage](./04-rag-agents/rag-retrieval-leakage.mdx) | vectorized ≠ anonymized, post-retrieval filtering leaves a window, cache / memory boundary bugs |
| **Cloud provider / co-tenant seeing** the prompt or weights | [Confidential inference](./05-frontier-deployment/confidential-inference.mdx) · [TEE](./01-foundations/trusted-execution-environment.mdx) · [HE·MPC](./01-foundations/he-mpc.mdx) | not verifying attestation = useless, root of trust is still the chip vendor + side channels, the crypto route is too expensive |
| The provider **retaining / training on / handing off** what you send | [Inference-service data boundary](./06-governance-compliance/inference-service-data-boundary.mdx) | "no training" ≠ "no retention", terms vary by endpoint / feature, a legal hold can override deletion |
| **Device data centralized**, or updates leaking training data | [Production-grade DP·FL](./05-frontier-deployment/dp-federated-learning.mdx) | FL ≠ private (updates leak), local DP's ε is often large, user-level vs sample-level |
| System prompt / conversation context / tool results extracted | [Context-surface privacy](./03-conversational-llms/context-surface-privacy.mdx) | instruction-style "don't recite" ≠ isolation, extraction detection is cat-and-mouse, the boundary must land on backend authn / secret management / permission-scoped retrieval |
| Private **PII** **reproduced** by the model in conversation | [PII regurgitation](./03-conversational-llms/pii-regurgitation.mdx) | scrubbing reduces ≠ eliminates (NER misses), de-identification ≠ de-association, output filtering is cat-and-mouse |
| The **model itself** cloned by queries / some parameters solved | [Model extraction & stealing](./01-foundations/model-extraction-stealing.mdx) | defense raises cost ≠ eliminates, stealing amplifies offline MIA / extraction, limiting output hurts usability |
| Deleted from the primary DB, but **copies still remain** in backups / logs / vector store / derived models | [Data lifecycle & deletion propagation](./06-governance-compliance/data-lifecycle-deletion.mdx) | propagation only covers known copies, still present within the backup expiry window, the derived-model copy needs unlearning |
| Wanting to **measure "how much I memorized"** before release | [Quantifying memorization & auditing](./02-memorization-extraction/quantifying-memorization.mdx) | a canary is a proxy not ground truth, high exposure ≠ necessarily extractable, memorization rises with scale so re-measure per version |

:::tip How to use this map
First locate the **threat** you worry about most, read that entry's **mechanism + buildable recipe + minimal testable assertions**, then jump laterally via each entry's "How this differs from neighboring techniques" to connect the whole line of defense. Every entry calls out the corresponding **false security** — don't just look at "what was used," look at "what it doesn't solve."
:::
