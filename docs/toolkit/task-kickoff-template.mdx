> 一句话摘要：开工前花两分钟把任务说清楚，比事后返工省得多。把下面这段填好发给我——它逼你交代清楚目标、验收标准、约束、相关文件、什么别碰，并要求我**先出计划、你点头再动手**。模糊的开场我只能猜，猜错的成本你来付。

把空填上，删掉用不到的行：

```text
# 目标
我要你完成的一句话目标：__________
为什么要做 / 它要解决什么问题：__________

# 验收标准（满足这些才算做完）
- [ ] __________
- [ ] __________
- [ ] 验证方式：跑 __________，应当看到 __________

# 约束（必须遵守）
- 安全：__________（如：不得硬编码密钥；外部输入要校验；涉及数据访问要鉴权）
- 依赖：__________（如：不要引入新依赖 / 只允许用已在 package 里的；不要升级版本）
- 代码风格：照着 __________（既有文件 / formatter / lint 规则）来，与周围代码保持一致

# 相关文件与上下文
- 要改的文件：__________
- 需要先读、用来对齐写法的参考文件：__________
- 已知的坑 / 背景（别人正在改 X、这块有历史原因）：__________

# 明确不在范围内（别碰）
- 不要动：__________（如：这些文件 / 这个模块 / 命名 / 格式 / 无关重构）
- 不要做：__________（如：不要顺手优化；不要重写整个文件；不要改测试以外的行为）

# 工作方式
先出计划：列出你打算改哪些文件、每处做什么、为什么，**等我确认后再写代码**。
计划阶段如有歧义或缺信息，先问我，别自己拿主意填空。
```

## 怎么用

- **目标 + 验收标准是地基**，别省。我没有验收标准时只能自己脑补一个「做完」的定义，往往和你想的不一样（见 [把模糊当成清晰：你以为说清了，其实只有你懂](../02-requirements/vague-as-clear.mdx)）。
- **「明确不在范围内」这一段最被低估**。我对「别动 X」这类显式约束是听得进去的；你不写，我就按默认的「顺手做好」自由发挥，diff 一膨胀就难评审。把不想要的东西点名禁掉，比事后挑出来便宜。
- **「先出计划、确认再动手」是省钱的关键一步**。让我先把意图摊开，你能在我写出错方向的代码之前就拦下来；Claude Code 的计划模式正是干这个的（见 [跳过计划模式：让我直接上手，等于让我边猜边改](../00-setup-collaboration/skipping-plan-mode.mdx)）。
- **不必每条都填**。小改动留「目标 + 验收 + 别碰 + 先计划」就够；越大越不可逆的任务，越值得把约束和上下文填满。
